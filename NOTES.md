## Розділ 1. Початок роботи

### 1. Встановлення pnpm
Встановила pnpm глобально через npm.

Команда:
npm install -g pnpm

У консолі відобразилось повідомлення:

- 1 package is looking for funding

![Install pnpm](./screenshots/photo_2026-03-02_16-06-13.jpg)


### 2. Створення проєкту
Створила проєкт через create-next-app з прикладом dashboard.

Команда:
npx create-next-app@latest nextjs-dashboard --example "https://github.com/vercel/next-learn/tree/main/dashboard/starter-example" --use-pnpm


Під час створення:
- було встановлено 202 пакети
- автоматично ініціалізовано Git репозиторій

Процес завершився повідомленням:
`Success! Created nextjs-dashboard`

![Create app](./screenshots/photo_2026-03-02_16-35-14.jpg)


### 3. Запуск dev-сервера
Перейшла в папку проєкту:
cd nextjs-dashboard

Виконала:
pnpm dev

Сервер запустився успішно:

## Розділ 5. Навігація між сторінками

### 1. Використання компонента Link

Компонент `Link` дозволяє виконувати навігацію на стороні клієнта без повного перезавантаження сторінки.

Для цього потрібно імпортувати `Link` з `next/link`.

Файл:

```
/app/ui/dashboard/nav-links.tsx
```

Після заміни тегу `a` на тег `Link` переходи між сторінками відбуваються без повного оновлення сторінки, що робить застосунок швидшим та більш схожим на нативний додаток.


### 2. Попереднє завантаження сторінок

У production режимі Next.js використовує prefetching.

Коли компонент `Link` з’являється у видимій частині браузера:

- автоматично попередньо завантажує код сторінки
- робить це у фоновому режимі

Тому коли користувач натискає посилання:

- код сторінки вже завантажений
- перехід відбувається майже миттєво

## Розділ 7. Отримання даних

У цьому розділі було розглянуто способи отримання даних у застосунку Next.js та створено сторінку огляду Dashboard.

### Отримання даних для Dashboard

Було реалізовано отримання даних для таких компонентів:

- **RevenueChart** — відображає графік доходів за останні 12 місяців.
- **LatestInvoices** — показує 5 останніх рахунків.
- **Card** — відображає статистику:
  - загальна сума оплачених рахунків
  - сума рахунків, що очікують оплати
  - кількість рахунків
  - кількість клієнтів


### Оптимізація запитів

Було розглянуто поняття **waterfall requests** (каскадні запити), коли кожен запит виконується після завершення попереднього.

Для покращення продуктивності використовується **паралельне отримання даних** за допомогою:

```javascript
Promise.all()
```

Це дозволяє запускати декілька запитів до бази даних одночасно.

### Результат

Було створено сторінку Dashboard, яка відображає:

- статистику рахунків
- графік доходів
- список останніх рахунків

## Розділ 8. Статичний та динамічний рендеринг
У цьому розділі було розглянуто принципи **статичного та динамічного рендерингу** в Next.js та їх вплив на продуктивність застосунку.

### Моделювання повільного отримання даних

Для демонстрації повільного отримання даних у файлі:

```
/app/lib/data.ts
```

було додано штучну затримку:

```javascript
await new Promise((resolve) => setTimeout(resolve, 3000));
```

Це імітує повільний запит до бази даних.

Після запуску застосунку можна побачити у консолі:

```
Fetching revenue data...
Data fetch completed after 3 seconds.
```

Це показує, що сторінка завантажується повільніше, оскільки динамічний рендеринг залежить від швидкості отримання даних.

### Результат

Було змодельовано повільне отримання даних та продемонстровано, як це впливає на завантаження сторінки Dashboard.

## Розділ 10. Додавання пошуку та нумерації сторінок

У цьому розділі було реалізовано пошук та пагінацію на сторінці **Invoices** у застосунку Next.js.

### Виконана робота

- Додано компонент **Search**, який дозволяє користувачам шукати рахунки-фактури.
- Реалізовано оновлення URL-параметрів за допомогою API Next.js:
  - `useSearchParams`
  - `usePathname`
  - `useRouter`
- Передано параметри пошуку до серверного компонента **Table**, щоб фільтрувати дані.
- Додано **debounce** (бібліотека `use-debounce`), щоб уникнути запиту до бази даних при кожному натисканні клавіші.
- Реалізовано **пагінацію** для перемикання між сторінками рахунків-фактур.
- Використано параметри URL (`query`, `page`).
